# AGENTS.md（Project Context）模板

在首次创建目标项目的 `AGENTS.md` 时使用此模板。后续更新时在现有内容基础上修改。

---

以下是模板内容：

```markdown
---
project_name: "<项目名>"
last_updated: "YYYY-MM-DD"
updated_by: ""
---

# Project Context

## 1. 技术栈

- 语言: ...
- 框架: ...
- 构建工具: ...
- 测试框架: ...
- 其他关键依赖: ...

## 2. 质量检查命令

- 类型检查: `<命令>`
- Lint: `<命令>`
- 测试: `<命令>`
- 构建: `<命令>`

## 3. 关键约束

<!-- 影响所有开发决策的硬性限制 -->
<!-- 例如：性能要求、兼容性要求、安全约束、运行环境限制 -->

- ...

## 4. 关键实现规则

<!-- 只写 agent 可能猜不到的、项目特有的约定 -->
<!-- 不要写通用最佳实践（如"使用有意义的变量名"） -->
<!-- 好的规则："所有 API 调用必须使用 apiClient 单例" -->
<!-- 坏的规则："使用 TypeScript"（agent 从文件后缀就能看出来）-->

**代码组织:**
- ...

**错误处理:**
- ...

**命名约定:**
- ...

## 5. 目录结构

<!-- 项目文件组织和每个目录的职责 -->

src/
├── ...

## 6. 已完成功能

<!-- 在每次 PR review 通过后更新 -->

| 功能 | 完成日期 | 对应 Plan |
|------|---------|-----------|

## 7. 已知限制与技术债务

<!-- 后续 task 需要注意的问题 -->
<!-- 例如："TASK-005 中临时用了 mock 数据，后续需要替换为真实 API" -->

- ...
```

## 更新原则

- **Focus on the unobvious**: 只写 agent 从代码片段中推断不出来的信息
- **Keep it lean**: 这个文件会被多个 agent 和 workflow 加载，太长浪费 context
- **反映现实**: 基于代码的实际状态更新，不是基于计划中的设计
- **第 6 节是动态维护的**: 每次 PR 合并后追加已完成功能
- **第 7 节传递跨 plan 信息**: 让后续 Monkey agent 知道哪些临时方案需要注意
